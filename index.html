文本编码能做什么、不能做什么

Base64、URL 转义与 Hex 都是编码：它们把字节改写成安全字符集，让文本与二进制数据能安全地经由邮件、URL 和 JSON 传输。它们对任何人都完全可逆，不提供任何保密性。用它们来安全地搬运数据，而不是隐藏数据。

Base64 的原理

Base64 由 RFC 4648 定义。它把输入按 6 位分组，每组映射到 64 个字符之一（A–Z、a–z、0–9，外加 + 和 /），末尾用 = 填充。每 3 个输入字节变成 4 个输出字符，因此编码后的文本比输入大约大 33%。URL 安全变体把 + 和 / 换成 - 和 _，使结果可安全放进链接。

编码不是加密

一个常见误区是把 Base64 当成保护数据的手段。它不是：任何人都能解码。本工具里的口令加密加入了一个密钥，但简单异或只是轻度混淆，挡不住有心的攻击者。要真正保密，请使用具备真正端到端加密的通讯工具，例如 Telegram 的 Secret Chat（密聊），它采用 MTProto 协议与 256 位 AES。

FAQ

我的文本会被上传吗？

不会。全部编码、解码与加密都在你的浏览器内用标准 Web API 完成，文本绝不离开你的设备。

Base64 会加密我的数据吗？

不会。Base64 是编码而非加密——任何人都能解码。它只是让字节便于传输，并不保密。

为什么 emoji 或中文的 Base64 和别的工具不一样？

因为本工具通过 TextEncoder 编码正确的 UTF-8 字节。直接调用 btoa() 的工具会在 Latin1 以上的字符上出错或损坏。

URL 编码和 Base64 有什么区别？

URL 编码（encodeURIComponent）用 % 序列转义在 URL 中不安全的字符；Base64 则把原始字节改写成 64 字符字母表。它们解决的是不同的传输问题。

这个口令加密安全吗？

不安全。它只是用于轻度混淆的简单异或。要真正保密，请使用具备真正端到端加密的通讯工具，如 Telegram 的 Secret Chat。

Base64 末尾的 = 是什么？

它是填充。Base64 以每 4 个字符为一块；当输入长度不是 3 字节的整数倍时，用 = 补满最后一块。
