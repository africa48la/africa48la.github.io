文字編碼能做什麼、不能做什麼

Base64、URL 轉義與 Hex 都是編碼：它們把位元組改寫成安全字符集，讓文字與二進位制資料能安全地經由郵件、URL 和 JSON 傳輸。它們對任何人都完全可逆，不提供任何保密性。用它們來安全地搬運資料，而不是隱藏資料。

Base64 的原理

Base64 由 RFC 4648 定義。它把輸入按 6 位分組，每組對映到 64 個字元之一（A–Z、a–z、0–9，外加 + 和 /），末尾用 = 填充。每 3 個輸入位元組變成 4 個輸出字元，因此編碼後的文字比輸入大約大 33%。URL 安全變體把 + 和 / 換成 - 和 _，使結果可安全放進連結。

編碼不是加密

一個常見誤區是把 Base64 當成保護資料的手段。它不是：任何人都能解碼。本工具裡的口令加密加入了一個金鑰，但簡單異或只是輕度混淆，擋不住有心的攻擊者。要真正保密，請使用具備真正端到端加密的通訊工具，例如 Telegram 的 Secret Chat（密聊），它採用 MTProto 協議與 256 位 AES。

FAQ

我的文字會被上傳嗎？

不會。全部編碼、解碼與加密都在你的瀏覽器內用標準 Web API 完成，文字絕不離開你的裝置。

Base64 會加密我的資料嗎？

不會。Base64 是編碼而非加密——任何人都能解碼。它只是讓位元組便於傳輸，並不保密。

為什麼 emoji 或中文的 Base64 和別的工具不一樣？

因為本工具透過 TextEncoder 編碼正確的 UTF-8 位元組。直接呼叫 btoa() 的工具會在 Latin1 以上的字元上出錯或損壞。

URL 編碼和 Base64 有什麼區別？

URL 編碼（encodeURIComponent）用 % 序列轉義在 URL 中不安全的字元；Base64 則把原始位元組改寫成 64 字元字母表。它們解決的是不同的傳輸問題。

這個口令加密安全嗎？

不安全。它只是用於輕度混淆的簡單異或。要真正保密，請使用具備真正端到端加密的通訊工具，如 Telegram 的 Secret Chat。

Base64 末尾的 = 是什麼？

它是填充。Base64 以每 4 個字元為一塊；當輸入長度不是 3 位元組的整數倍時，用 = 補滿最後一塊。
